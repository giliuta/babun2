## 1. TL;DR

Babun2 — рабочая Next 16 + React 19 PWA на 62 страницах + 9 API-роутов, multi-tenant Supabase backend (35+ миграций, 2 edge-functions), монорепо `apps/web` + `apps/mobile` (заглушка) + `packages/shared`. Кодовая база богатая и логически связная: финансовая абстракция централизована (`computeFinancials`), middleware простой и корректный, есть SSRF-защита, Stripe и Twilio-вебхуки с HMAC-проверкой, Supabase-миграции упорядочены по датам. Production-build чистый, `tsc --noEmit` чистый, lint — 51 «error» и 90 warnings, но это сообщения React-Compiler (setState-in-effect, refs-in-render) — не блокируют сборку, но указывают на потенциальный rerender-хвост на iOS. Главные проблемы: **(1)** 5 из 9 vitest-файлов не запускаются вообще (vitest-alias не покрывает subpath-exports `@babun/shared/local/...`), **(2)** в `next.config.ts` зарегистрированы редиректы на 7 несуществующих страниц (`/expenses`, `/payroll`, `/reports`, `/brigades`, `/schedule`, `/route`, `/waitlist`) — короткие URL ведут в 404, **(3)** золотое правило CLAUDE.md «400 строк на компонент» нарушено в нескольких местах (dashboard/page.tsx — 1613 строк), **(4)** в репозитории ~50 untracked-мусорных файлов с пустыми именами и `babun-crm/supabase/migrations/001_initial_schema.sql` живёт параллельно с актуальным каталогом миграций в `apps/web/supabase/`. Финансовая Phase 1+2 действительно работает, но «отдельные страницы expenses/payroll/reports» свёрнуты в табы внутри `/dashboard/finances` — расхождение с тем, как они описаны в задании.

## 2. Что работает хорошо

1. **Production-build зелёный.** `npm run build` (Turbopack) успешно собрал 60+ маршрутов, 18 секунд на статической генерации, итоговый `.next/` — 44 MB (37 MB server + 3.8 MB static). Размер вменяемый.
2. **`npx tsc --noEmit` чистый** во всём `apps/web/` — strict-режим выдержан, никаких `any`-кейсов на верхнем уровне (за исключением ~40 точечных `as any` для Supabase RPC, см. P3).
3. **Финансовая абстракция вычищена.** `packages/shared/src/local/finance/compute.ts` — единая точка для income/expense/profit/cashbox с явными правилами «всё в евро-целых, cents → eur на входе». Это решает проблему расхождения вкладок, упомянутую в Phase 1.
4. **Тесты на финансовую логику есть и проходят.** `__tests__/finance/expenses.test.ts`, `compute.test.ts`, `notifications.test.ts` — 36 тестов passed, плюс sync/format и billing/quota и stripe-webhook.
5. **Миграции localStorage идемпотентны.** `apps/web/src/lib/migrations/index.ts` ведёт `babun2:migrations` ключ, прогоняет 0001-0004 ровно один раз каждую.
6. **Supabase-миграции упорядочены.** 35 SQL-файлов в `apps/web/supabase/migrations/` с ISO-дат-префиксом, охватывают auth, RLS, onboarding, billing, push, SMS, admin, brand, GDPR-export, calendar settings.
7. **Безопасность API.** `/api/resolve-map-link` — SSRF-hardened (allowlist хостов + IP-блоклист), `/api/stripe/webhook` и `/api/twilio/status` — HMAC-verify, `/api/account/delete` каскадно вычищает tenant-данные.
8. **Middleware простой и грамотный.** `src/middleware.ts` — только проверка наличия cookies без декода JWT (комментарий объясняет, что попытка декодить exp ломала refresh-loop на iOS PWA в v420).
9. **Финансовые типы строго в cents.** `packages/shared/src/db/types/finance.ts` — Brigade, ServiceLine, AppointmentFinance, FinancePayment, Expense, Payroll — всё в `*Cents`/`*Eur` (cents) с явными правилами расчёта.
10. **Lazy-loading модалок в Dashboard.** `dashboard/page.tsx` грузит CityPickerModal, DayFinanceModal, SpecialScheduleModal, RepeatCopyModal, AppointmentSheet, PersonalEventSheet через `dynamic({ ssr: false })` — это удерживает initial-bundle разумным.

## 3. Что не работает (P0)

### P0-1. Битые редиректы → 404
В `babun-crm/apps/web/next.config.ts:73-99` зарегистрированы редиректы со старых коротких URL на `/dashboard/<x>`, но семь целевых страниц физически НЕ существуют:

| Источник в `next.config.ts` | Цель | Реальность |
|---|---|---|
| `/expenses` | `/dashboard/expenses` | **404** — папки `dashboard/expenses/` нет |
| `/payroll` | `/dashboard/payroll` | **404** — папки нет |
| `/reports` | `/dashboard/reports` | **404** — папки нет |
| `/brigades` | `/dashboard/brigades` | **404** — папки нет |
| `/schedule` | `/dashboard/schedule` | **404** — папки нет |
| `/route` | `/dashboard/route` | **404** — папки нет |
| `/waitlist` | `/dashboard/waitlist` | **404** — папки нет |

Финансовые экраны фактически живут как **табы внутри `/dashboard/finances`** (`Mode = "income" | "expenses" | "summary" | "debts" | "payroll"`). Описание Phase 2 в задании («`/dashboard/expenses`, `/dashboard/payroll`, `/dashboard/reports`, `/dashboard/brigades`») не соответствует реальной маршрутизации.

Влияние: пользователь по букмарку или истории попадает на белую 404. Простое исправление — либо убрать пять-семь редиректов, либо перенацелить их на `/dashboard/finances?tab=...`.

### P0-2. 5 из 9 vitest-файлов не запускаются вообще
`npx vitest run` отчитался: `5 failed | 4 passed (9)` — но **36 тестов passed, 0 failed**, потому что в 5 файлах vitest падает ещё до коллекции тестов на `Failed to resolve import "@babun/shared/local/payroll"`, `reconciliations`, `notifications`, `expenses`. Эти модули **физически существуют** (`packages/shared/src/local/payroll.ts`, `reconciliations.ts`, и т.д.), но `apps/web/vitest.config.ts:14-19` маппит только `@babun/shared` целиком на корень пакета — subpath-exports не доходят.
