# Babun2 — Ideas beyond roadmap

**Дата:** 2026-04-19
**Контекст:** дополняет `roadmap.md` и `UX_AUDIT.md`. Ничего, что уже в STORY-001…010 (Supabase, Auth, online booking, WhatsApp inbox, Stripe, AI marketing, route optimization) — сюда не входит.
**Persona:** диспетчер AirFix на скутере +35 °C, одна рука; 2 бригады, 903 клиента; HVAC-выезды, повторные визиты раз в 1–3 года.

---

## 1. Automation / AI-assisted

### 1.1 AC-unit fingerprint (S)
Бригадир фоткает шильдик кондиционера → Claude Vision извлекает `brand / model / serial / capacity`. Запись в `appointments.equipment[]`, автодополнение при повторном визите.
**Pilot:** неделя, 20 записей; измерить долю успешного parse vs ручной ввод.

### 1.2 Post-visit summary для SMS/WhatsApp (M)
После `status=completed` Claude читает `comment + services[] + photos[].caption` и генерирует 2-строчное RU-резюме «что сделали, что требует внимания». Диспетчер подтверждает → отправка. Повышает NPS без писанины.
**Pilot:** 10 закрытий, оценка владельца «отправил бы?» (y/n).

### 1.3 Repeat-service predictor (M)
Rule-engine (без LLM): по `services[].category + client.locations + last_visit_date` генерит задачу «через 11 мес предложить чистку». Падает в новый «Pipeline»-экран, не в календарь. HVAC-специфика: чистка раз в год, газ раз в 2–3 года.
**Pilot:** прогнать по 903 клиентам, показать владельцу топ-50 кандидатов на апрель.

### 1.4 Smart dispatcher (L)
Claude + rules предлагают 3 слота при создании записи: учитывает `TeamSchedule`, текущую загрузку дня, город клиента vs уже запланированные в тот день выезды в этом городе, пробки (Google). UI: чипы «Вт 10:00 · Brigade1 · +2 рядом в Limassol». Не автомат — предложение.
**Pilot:** 30 новых записей, замер «сколько раз выбрали первый чип».

## 2. Field-crew features

### 2.1 Before/after gallery с подписью и гео (S)
`Appointment.photos[]` уже есть в типе (`lib/appointments.ts:114`), но UI нет. Две зоны «До / После», EXIF → геотег, голосовая подпись (Web Speech API, RU). Клиенту отправляется ссылка на галерею.
**Pilot:** 5 выездов, проверить что обе зоны заполнены, метаданные сохранены.

### 2.2 Dynamic checklist по категории услуги (M)
Для каждой `service.category_id` — свой JSON-шаблон чек-листа («проверить давление», «промыть фильтр»). Галочки сохраняются на записи. Владелец видит % завершённости в отчётах.
**Pilot:** 2 категории (Installation, Maintenance), 20 записей, замер пропущенных пунктов.

### 2.3 Client signature + auto-invoice PDF (M)
Canvas 2D подпись → блоб на запись → PDF с итогом (услуги, материалы, сумма, подпись) → share-sheet в WhatsApp. Без Stripe — только документ. Легитимизация на Кипре (VAT-ready).
**Pilot:** 10 завершённых записей, проверить валидность PDF на iPhone + Android.

### 2.4 Offline queue (L)
LocalStorage → IndexedDB + outbox. Любое изменение записи во время отсутствия сети помечается `pending_sync`. При возврате — автосинхрон, конфликты решаются last-write-wins с уведомлением. Критично для горных деревень и подземных гаражей.
**Pilot:** airplane-mode тест: 3 изменения оффлайн, 1 конфликт, проверить что ничего не потерялось.

## 3. Owner / analytics

### 3.1 Unit economics dashboard (S)
Один экран, 4 KPI: средний чек, цена рабочего часа бригады, gross-margin по услуге, конверсия лид→запись. Все — уже считаемо из существующих типов. Заменяет мутный `finances` + `reports`.
**Pilot:** показать владельцу за 1 день, собрать 3 уточнения по формулам.

### 3.2 Cohort retention heatmap (M)
Когорты по месяцу первого визита × месяцы после. Цвет = % клиентов с повторным заказом. HVAC-бизнес живёт на ретеншене — сейчас невидимо. Цифра «доля пришедших из 2024 в 2026» должна быть на виду.
**Pilot:** импорт 903 клиентов → рендер таблицы, владелец называет «подозрительные когорты».

### 3.3 Seasonality + capacity alert (M)
Скользящее среднее загрузки бригад по неделям → при прогнозе >90% за 7 дней вперёд — алерт «бронирования мая перекрывают capacity, найми помощника / подними цену». Rule-engine на исторических данных.
**Pilot:** прогон на Bumpix-экспорте 2023–2025, оценка «попал бы алерт» для реальных жарких недель.

### 3.4 Profitability leaderboard: услуга × бригада (S)
Таблица `service × team` с маржей (revenue − labor − material). Сразу видно что «Brigade 2 + Installation» — убыточная комбинация.
**Pilot:** за месяц данных, обсудить с владельцем «убрать бы» кандидатов.

## 4. Client-side

### 4.1 Share-link на визит (S)
Уникальный URL `/v/{token}` при создании записи: дата, время, имя мастера, телефон, карта (Google Maps deep-link). Клиент делится с охраной ЖК или супругой. Аналог калькомовской ссылки на встречу.
**Pilot:** 20 созданий → SMS-шаблон с ссылкой, замер open-rate.

### 4.2 Self-reschedule ≥24ч (S)
Та же share-link: кнопка «Перенести» доступна если до визита >24 ч. Показывает 3 ближайших свободных слота той же бригады. Без полноценного online-booking, только для уже созданных записей.
**Pilot:** у 10 клиентов включить ссылку, замер «перенесли сами vs позвонили».

### 4.3 Mini-portal (M)
`/my/{phone-token}`: история визитов, фото до/после, гарантии (дата+срок, вычисляется из service), кнопка «Повторить такой же заказ». Авторизация — OTP по SMS.
**Pilot:** вручную включить 5 «VIP» клиентам, снять отзывы.

### 4.4 Post-visit review funnel (S)
Через 2 часа после `status=completed` SMS с коротким опросом (1–5). При 5★ → Google Reviews link. При 1–3★ → форма с полем «что улучшить» → прямо в `chats/` владельцу. Rule: один раз на клиента в 90 дней.
**Pilot:** 30 визитов, замер звёзд на Google + количество жалоб, перехваченных до отзыва.

## 5. Platform → multi-vertical SaaS

### 5.1 Vertical pack = JSON-overlay (M)
Поверх базового Babun2 — `vertical.json` с `default_services / checklists / sms_templates / form_fields / default_categories / color_kinds`. Пять стартовых: HVAC (AirFix), Barber, Cleaning, Auto service, Phone repair. Переключение — при создании tenant'а. Никаких отдельных codebase.
**Pilot:** сгенерировать Barber-pack, установить второй tenant на проде, пройти full-flow: создать услугу «Стрижка 20€», запись, закрытие.

### 5.2 Client-identity plug (S)
Разные вертикали хотят разные обязательные поля клиента. Barber: `birthday`. Auto: `plate_number + vin`. Phone repair: `imei`. Реализация — расширяемые `client.custom_fields[]` на vertical pack.
**Pilot:** включить Auto-pack → поле «Госномер» обязательное при создании клиента → проверить что HVAC-теnant это не затронуло.

### 5.3 Pricing: per-brigade, not per-seat (M)
2 бригады = €49/мес, 4 бригады = €89, unlimited = €149. Стимулирует рост: владелец добавляет бригаду не для «ещё одного сиденья», а для capacity. В отличие от per-seat (Jobber, Housecall Pro) не штрафует за помощников с view-only доступом.
**Pilot:** симуляция: посчитать что бы AirFix платил сейчас и через год; сравнить с Jobber Growth ($169).

### 5.4 Bring-your-own-channel (L)
Вертикалям важны разные каналы. HVAC/Cleaning — WhatsApp+SMS. Barber — Instagram DM. Auto — Telegram. Messenger-adapter interface: одна абстракция `send(phone, text)`, конкретный tenant выбирает подключённые каналы в настройках. Строим после STORY-005 (WhatsApp).
**Pilot:** после WA — добавить Telegram-adapter, показать что Barber-tenant по умолчанию предлагает Instagram+Telegram без WA.
