# Sprint 001 — Product Validation: Before / After Photos

Persona: HVAC-бригада AirFix, Cyprus, +35°C, одна рука на телефоне, вторая в кондиционере.

## 1. Доменная валидация — реальные use-cases

- **Чистка фильтров / внутреннего блока (60% визитов).** Фото «до» = чёрная пыль/плесень; «после» = белый фильтр. Это главный ROI фичи — доказательство работы и anti-claim материал.
- **Установка наружного блока.** «До» = голая стена, «после» = блок + кронштейн + аккуратные трассы. Фото уходит в Instagram/WhatsApp как маркетинг.
- **Ремонт платы / замена конденсатора.** «До» = сгоревший компонент крупным планом (доказательство неисправности клиенту перед оплатой), «после» = новая деталь. Без фото клиент не верит.
- **Протечка / забитый дренаж.** Фото лужи на полу = страховой случай, доказательство причины.
- **Приём объекта (реже).** Фото царапины на стене/мебели до начала работ — защита от «вы сломали».

**Вывод:** фича валидна. Частота ~2-6 фото на визит, уже сегодня бригады это делают в нативной камере iPhone и вручную шлют в WhatsApp — фича перехватывает поток в CRM.

## 2. Минимальный набор `kind` — только **3**

- `before` · `after` · `other` — достаточно для MVP.
- Соблазны отклонить: «серийный номер», «пломба», «счёт», «повреждение» — всё это лезет в `other` + свободный caption. Лишние категории = лишние тапы. Расширим, когда увидим как реально фоткают (аналитика по caption'ам через 2 месяца).

## 3. Подпись (caption) — **auto, не ручная**

- Бригада **не будет** печатать caption в +35°C с мокрыми пальцами. Проверено в Bumpix — поле оставалось пустым в 90% случаев.
- Формат auto-caption: `«До · 14:35»` (kind + HH:MM). Локации/комнаты опускаем — бригада не будет выбирать «спальня/зал».
- Ручной ввод — опциональный, спрятан за длинным тапом на фото. Не блокирует flow.

## 4. Кто увидит фото — только AppointmentSheet (MVP)

- **MVP:** фото видно только внутри конкретной записи. Этого хватает.
- **Структура `photos[]` на записи не блокирует будущее:**
  - Профиль клиента → агрегация `client.locations[*].appointments[*].photos` — делается на read-side без миграции.
  - PDF-отчёт / share-link клиенту → читает те же `photos[]`, добавит только публичный токен.
  - Instagram-маркетинг → экспорт pack через selection — тоже поверх существующей структуры.
- **НЕ делаем в MVP:** cross-appointment галерею клиента, публичный share-link, WhatsApp-отправку.

## 5. Privacy / юридика

- В кадр попадает интерьер клиента (дети, документы на столе, лица). Это чувствительно даже в B2C на Cyprus.
- **MVP-минимум:**
  - Один раз при первом фото в сессии показать toast: «Фото сохраняется локально. Спросите клиента перед съёмкой».
  - `consent_given: boolean` на уровне **appointment** (не на фото) — галочка «клиент разрешил фото» в PhotoBlock, дефолт `false`. Если false — фото всё равно сохраняется, но помечено, и в будущем share/отчёт этот флаг блокирует публикацию.
  - Никаких EXIF с GPS в base64 (canvas.toDataURL EXIF режет — проверить).
- **НЕ делаем:** отдельную подпись клиента, юр. форму согласия, GDPR export по фото. Это после Supabase.

## 6. Что НЕ делать в MVP (явный список соблазнов)

- Серверная загрузка / Supabase Storage — только localStorage base64.
- Серверное пережатие / thumbnail generation — всё client-side (canvas 0.6 jpeg).
- OCR серийных номеров, AI-описание «что на фото», auto-tagging before/after по ML.
- Видео, live-photo, burst mode.
- Reordering, rotation, crop, filters, аннотации стрелками.
- Share-link клиенту, WhatsApp-кнопка, email-send.
- Cross-appointment галерея (профиль клиента, timeline).
- Лимит на размер сверх capped 20 фото/запись — не нужен UI, просто FIFO-вытеснение.

## 7. Success-метрики (для след. sprint'а)

- ≥60% completed-записей имеют ≥1 фото через 2 недели.
- ≥30% имеют пару before+after (главный use-case).
- Средний caption-edit-rate <15% (подтверждает что auto-caption достаточно).
